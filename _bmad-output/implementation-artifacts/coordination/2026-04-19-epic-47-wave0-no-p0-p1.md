## Agent Outputs (Detailed)

### MS-47-A1 Result — Contract Freeze (`@bmad-architect`)

#### Canonical storage and API contract
- **Storage authority:** `settings_strings` with:
  - `company_id = auth.company_id`
  - `outlet_id = NULL`
  - `setting_key = ap_reconciliation_account_ids`
  - `setting_value = JSON array of account IDs`
- **Config endpoint (proposed):** `PUT /api/accounting/ap-reconciliation/settings`
- **Summary endpoint (proposed):** `GET /api/accounting/ap-reconciliation/summary?as_of_date=YYYY-MM-DD`
- **Validation constraints:**
  - `account_ids`: integer array, unique, min 1, max 50
  - `as_of_date`: strict `YYYY-MM-DD`

#### Tenant ownership rules for account IDs
Every configured `account_id` must satisfy:
1. account exists
2. account belongs to authenticated `company_id`
3. account active/not deleted
4. account AP-control compatible (`is_payable=1` and/or liability/creditor type)

Validation must be all-or-nothing; any invalid ID rejects entire request.

#### Compatibility and absent-setting behavior
- If `ap_reconciliation_account_ids` absent:
  - attempt explicit compatibility bridge to `company_modules.purchasing_default_ap_account_id`
  - only if account passes ownership/type checks
  - expose response source as `purchasing_default_ap_account_id`
- If unresolved, fail closed with explicit error (e.g., `409 AP_RECONCILIATION_SETTINGS_REQUIRED`).

#### A1 risk register
- **P0:** wrong FX math in reconciliation (`base != original * rate`) → integration assertions mandatory
- **P1:** cross-tenant account leakage → strict `company_id` validation on write + read paths
- **P1:** implicit fallback ambiguity → explicit source + fail-closed when unset

#### A1 status
- **Go:** ✅ Conditional go (no unresolved P1 if above rules enforced)

### MS-47-A2 Result — Schema/Migration Plan (`@bmad-architect`)

#### Required schema (minimum)
1. **`fiscal_periods`** (required for Story 47.5; also supports 47.1/47.6 cutoff/period logic)
   - tenant column `company_id` mandatory
   - period window columns and status required
   - date/status indexes for closed-period checks
2. **`supplier_statements`** (required for Story 47.3)
   - unique `(company_id, supplier_id, statement_date)`
   - currency + closing balance + status
   - statement date indexes
3. **`ap_exceptions`** (required for Story 47.4)
   - company-scoped exception records with type/status/source references
   - deterministic `exception_key` + unique `(company_id, exception_key)` for idempotent detection
   - worklist indexes (status, detected_at, type, due_date)

#### Migration sequencing
1. Create `fiscal_periods`
2. Create `supplier_statements`
3. Create `ap_exceptions` (after statements/periods if FK-linked)

#### Migration portability strategy
- Use guarded DDL via `information_schema` checks
- Avoid non-portable `ADD COLUMN IF NOT EXISTS`
- Keep rerunnable/additive migrations only

#### A2 risk register
- **P1:** missing tenant columns/FK scoping causes leakage risk
- **P1:** non-rerunnable DDL breaks deploy safety
- **P1:** non-idempotent exception generation without unique key
- **P2:** timezone/date indexing mismatch hurts query stability/perf

#### A2 status
- **Go:** ✅ Conditional go (no unresolved P1 if schema + guarded migrations applied)

### MS-47-A3 Result — ER + Temporal/Immutability (`@bmad-analyst`)

#### Relationship findings
- Epic 47 spans AP transactions (invoice/payment/credit), GL journals, period close, supplier statements, exceptions, and reconciliation snapshots/audit trail.
- **Blocker identified:** missing `fiscal_periods` dependency for period-close controls.

#### Locked vs mutable matrix
- **Locked at posting/finalization:** transaction amounts, exchange rate used, GL mapping, supplier reference, transaction date.
- **Mutable (forward-only effect):** reconciliation account-set config, tolerance thresholds, guardrail policy.
- Config changes must never retroactively alter posted/journalized results.

#### Temporal/cutoff decisions
- Timezone precedence: `outlet.timezone` → `company.timezone`; no UTC fallback.
- Inclusion rule: `<= as_of_date` means through end of local business day.
- Normalize in application constants, avoid wrapping indexed columns with SQL functions.

#### Ambiguities to clarify before Wave 1
- Exact exchange-rate effective window semantics for reconciliation.
- Outlet-scoped timezone behavior in AP reconciliation flows.
- Snapshot versioning and retention policy.

#### A3 risk register
- **P0:** missing fiscal periods blocks closed-period enforcement story
- **P1:** timezone ambiguity yields off-by-one-day reconciliation drift
- **P1:** mutable config affecting historical interpretation without snapshot lock

#### A3 status
- **Go:** ✅ Conditional go with blocker resolution (`fiscal_periods`)
