**3. Multi-Tenancy Architecture**

**4. Settings System**
- Typed settings registry (12 keys)
- Environment variable fallbacks
- Company + outlet scoped settings
- Audit logging for all changes

### What We Learned

**1. Auth Architecture Mature**
- JWT access tokens with configurable TTL
- Refresh token rotation prevents replay attacks
- Role hierarchy enforced (lower-level admins can't modify higher)
- Module-based permissions with bitmasks (create=1, read=2, update=4, delete=8)

**2. Database Design Patterns**
- Proper foreign key relationships
- Unique constraints for data integrity
- Indexing for query performance
- Soft deletes via is_active flags

**3. Integration Points**
- Auth guards reusable across all endpoints
- Company context available throughout request lifecycle
- Outlet access checks for multi-location businesses
- Permission checking at API boundaries

### Challenges

**1. Discovery Context**
- Foundation was built before project tracking began
- No clear documentation of what existed
- Had to reverse-engineer from codebase

**2. Super Admin Complexity**
- Special handling required for super admin
- Can only modify own account
- Bypasses normal permission checks
- Needs careful security review

**3. Settings Migration**
- Multiple schema versions for settings
- Backward compatibility requirements
- Environment variable overrides

## 📝 LESSONS LEARNED

### For Authentication Systems
1. **Token Rotation:** Refresh token rotation prevents replay attacks
2. **Password Hashing:** Auto-upgrade on login improves security over time
3. **Multi-Factor:** Foundation supports MFA addition
4. **Session Management:** Database-backed sessions allow revocation

### For Authorization Systems
1. **Bitmasks:** Efficient permission storage and checking
2. **Role Hierarchy:** Prevents privilege escalation
3. **Module Scoping:** Clean separation of feature access
4. **Outlet Context:** Multi-location businesses properly supported

### For Multi-Tenancy
1. **Company Isolation:** Clean data separation
2. **Outlet Scoping:** Hierarchical access within companies
3. **Foreign Keys:** Maintain referential integrity
4. **Unique Constraints:** Prevent duplicates per scope

### For Discovery Process
1. **Foundation First:** Must audit foundation before other epics
2. **Dependency Mapping:** Clear understanding of what depends on foundation
3. **Security Review:** Foundation requires careful security validation
4. **Documentation Gap:** Original implementation lacked documentation
