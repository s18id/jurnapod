# Jurnapod - Epic Breakdown

## Overview

This document provides the complete epic and story breakdown for Jurnapod, decomposing the requirements from the Architecture, ADRs, and technical initiatives into implementable stories.

## Requirements Inventory

### Functional Requirements

**FR1:** Type-safe database queries - The system must provide compile-time validation for SQL column names and table relationships to catch errors before runtime.

**FR2:** Kysely integration with existing DbClient pattern - The system must integrate Kysely as a type-safe query builder while preserving the existing `DbClient` interface pattern for dependency injection and testing.

**FR3:** Incremental migration path - The system must allow gradual migration of routes from raw SQL to Kysely without requiring a big-bang rewrite.

**FR4:** Raw SQL fallback for complex queries - The system must preserve the ability to use raw SQL for complex financial queries (GL posting, reconciliation, reports) where explicit SQL control is required.

**FR5:** Transaction support - The system must support database transactions with both Kysely and raw SQL approaches.

**FR6:** Connection pool reuse - The system must reuse the existing mysql2 connection pool for both Kysely and raw SQL queries.

**FR7:** Type generation from database schema - The system must auto-generate TypeScript types from the database schema to ensure type safety.

**FR8:** Manual type extensions - The system must allow manual type extensions for business logic types that cannot be auto-generated.

### Non-Functional Requirements

**NFR1:** Performance - Kysely queries must not introduce measurable performance overhead compared to raw SQL.

**NFR2:** Maintainability - The type-safe query builder must improve developer velocity for simple CRUD operations while preserving explicit SQL control for complex queries.

**NFR3:** Compatibility - All queries must remain compatible with MySQL 8.0+ and MariaDB.

**NFR4:** Auditability - SQL generated by Kysely must be readable and auditable for financial correctness.

**NFR5:** Test coverage - All migrated routes must pass existing unit tests and maintain test coverage standards.

**NFR6:** Type safety - The system must provide compile-time type checking for database queries without runtime overhead.

### Additional Requirements (from Architecture and ADRs)

**From ADR-0007 (MySQL2 Pool Singleton with Raw Parameterized SQL):**
- Pool singleton pattern must be preserved for connection management
- `dateStrings: true` configuration must be maintained for date handling
- Money values must remain as strings (DECIMAL) - never coerced to JavaScript number
- Transaction management pattern must be preserved

**From ARCHITECTURE.md:**
- Accounting/GL is the financial source of truth - all queries must preserve correctness
- POS is offline-first - sync operations must remain idempotent
- Multi-tenant isolation - all queries must enforce `company_id` and `outlet_id` scoping
- Immutable financial records - correction flows (VOID/REFUND) must be preserved

**From AGENTS.md:**
- Import path conventions: Use `@/` alias for imports from `apps/api/src/`
- Database pool cleanup in tests is mandatory
- All tests must pass after migration
- Type checking and lint must pass

### UX Design Requirements

N/A - This is a backend infrastructure initiative with no UI changes.

## Epic 0: Infrastructure & Technical Debt

**Goal:** Improve developer velocity and code quality through foundational infrastructure improvements that enable safer, faster development without compromising the explicit SQL control required for financial correctness.

**Business Value:**
- Reduce runtime SQL errors through compile-time type checking
- Improve developer velocity for simple CRUD operations (~30% faster for CRUD)
- Maintain explicit SQL control for complex financial queries
- Preserve auditability and correctness for accounting/GL operations

**Success Metrics:**
- **Compile-time error detection:** SQL column/table typos caught at compile time, not runtime
- **Developer velocity:** Measured reduction in time for CRUD route implementation
- **Test coverage:** All existing tests pass (473+ tests)
- **Type safety:** `npm run typecheck` passes with zero errors
- **No regressions:** All financial queries return identical results

**Technical Context:**
- Current approach uses raw SQL via `mysql2/promise` (ADR-0007)
- Custom `DbClient` interface pattern for dependency injection
- TypeScript type aliases extending `RowDataPacket`
- Zod schemas in `@jurnapod/shared` for validation contracts
- Complex financial queries require explicit SQL control

**Why Kysely (not Prisma):**
1. **N+1 Control**: Explicit JOINs, same mental model as raw SQL
2. **SQL Transparency**: Generated SQL is predictable and auditable
3. **Type Safety**: Compile-time column/table validation
4. **Incremental Adoption**: Can coexist with raw SQL
5. **No Magic**: Developer still writes the query logic

**Rollback Strategy:**
- Each story is independently deployable
- Raw SQL fallback is preserved throughout
- If critical issues arise, revert to raw SQL by removing Kysely queries
- No schema changes required - purely application layer

**Given** the DbClient interface with Kysely
**When** `npm run typecheck -w @jurnapod/api` is run
**Then** type checking passes with zero errors

**Technical Requirements:**

1. **DbClient Interface Extension:**
   ```typescript
   // packages/modules-accounting/src/accounts-service.ts
   import type { Kysely, DB } from '@jurnapod/db/kysely';

   export interface AccountsDbClient {
     // Preserve existing raw SQL methods
     query<T = any>(sql: string, params?: any[]): Promise<T[]>;
     execute(sql: string, params?: any[]): Promise<{ affectedRows: number; insertId?: number }>;
     
     // NEW: Type-safe query builder
     kysely: Kysely<DB>;
     
     // Preserve existing transaction methods
     begin?(): Promise<void>;
     commit?(): Promise<void>;
     rollback?(): Promise<void>;
   }
   ```

2. **MySQL Adapter Implementation:**
   ```typescript
   // apps/api/src/lib/accounts.ts
   import { createKysely } from '@jurnapod/db/kysely';

   class MySQLAccountsDbClient implements AccountsDbClient {
     private connection: PoolConnection | null = null;
     private _kysely: Kysely<DB> | null = null;

     get kysely(): Kysely<DB> {
       if (!this._kysely) {
         this._kysely = createKysely(this.pool);
       }
       return this.connection 
         ? this._kysely.withConnection(this.connection)
         : this._kysely;
     }
   }
   ```

### TD-001: N+1 Query Pattern in COGS Posting

**Location:** `apps/api/src/lib/cogs-posting.ts:484-501`

**Description:** Item account lookup loop causes N queries for N items in a sale.

**Impact:** A 50-item sale makes 50 separate queries to `items` + `company_account_mappings` tables.

**Recommendation:** Batch fetch all item accounts upfront:
```typescript
const itemIds = this.saleDetail.items.map(i => i.itemId);
const accounts = await getItemAccountsBatch(this.dbExecutor, itemIds);
```

**Priority:** P2 — **Resolved in Story 2.6** ✅

### TD-002: N+1 Query Pattern in COGS Calculation

**Location:** `apps/api/src/lib/cogs-posting.ts:171-235`

**Description:** Per-item inventory lookup loop causes up to 2N queries for N items.

**Impact:** A 100-item sale could make up to 200 queries.

**Recommendation:** Batch inventory and price lookups:
```typescript
SELECT product_id, SUM(quantity_delta), SUM(quantity_delta * unit_cost)
FROM inventory_transactions
WHERE company_id = ? AND product_id IN (?,?,?)
GROUP BY product_id
```

**Priority:** P2 — **Resolved in Story 2.7** ✅

### TD-003: N+1 Query Pattern in Recipe Composition

**Location:** `apps/api/src/lib/recipe-composition.ts:532, 710`

const ingredientIds = rows.map(r => r.ingredient_item_id);

**Business Value:**

- Preserve explicit SQL control for complex financial queries

**Technical Context:**

4. Validates Kysely patterns at scale
